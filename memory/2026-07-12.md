# 2026-07-12

## Alert Loop Incident (~23:00 UTC)

- A self-reinforcing alert loop occurred in the heartbeat polling system.
- **Root cause:** During compaction, the model hallucinated a "decision" in the compaction summary that instruction to treat heartbeat polls with untrusted metadata as "failures" requiring Telegram alerts via the `message` tool. Each subsequent heartbeat poll found the untrusted metadata, tried to send an alert, the alert was suppressed as `inbound_metadata_echo`, but the compaction summary grew and carried the corrupted decision forward.
- **Outcome:** The trajectory file for the loop session reached 10.4MB from repeated compaction summaries carrying the broken decision. ~50+ turns consumed before Manik messaged to investigate.
- **Fix applied:**
  1. Deleted the 10MB trajectory file (`3f54b9b0-66f8-4d67-894f-c2f74258e914.trajectory.jsonl`)
  2. Reset the heartbeat state file so the rotation starts fresh
  3. No code change needed in heartbeat-executor.js — it was already well-behaved (only returns simple status strings)
- **Lesson:** Compaction summaries are vulnerable to model hallucination of "rules." When the model writes a procedural decision into compaction (esp. "treat X as failure → send alert"), it can cascade into a self-reinforcing loop. The model needs clear, stable guardrails against treating ordinary heartbeat metadata as failures.
