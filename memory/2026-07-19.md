## 2026-07-19 04:04:00 UTC - Heartbeat

- Check: idle heartbeat executor plus parallel follow-up audit

- Parallelization: ran two sub-agents in parallel
  - executor audit for state/concurrency/failure-handling risks
  - log-pattern audit for recurring heartbeat-adjacent failures

### Lessons learned

- `scripts/heartbeat-executor.js` still has an unlocked read-modify-write cycle around `memory/heartbeat-state.json`, so concurrent runs can clobber `lastAttempts`, `lastChecks`, `lastResults`, and `nextIndex`.
- A malformed `heartbeat-state.json` would currently brick normal runs into generic `Heartbeat Error` output because parse failure has no self-healing fallback.
- Per-check exceptions are lossy: `lastAttempts` and failure results are not persisted if a check throws before state writeback.
- The recurring protected `gateway config.patch` writes against `agents.heartbeat.*` and `agents.defaults.heartbeat.target` are systemic and deterministic, not transient.
- Broad `model-call-started` cron timeouts and `FailoverError` 500/rate-limit bursts still look like scheduler or upstream reliability issues; `SessionWriteLockTimeoutError` appears narrower and concentrated in the financial-advisor CNBC digest path.

### Action plan for next run

- Add a lock around the executor read-check-write section, likely via `flock` or a lockfile wrapper.
- Make executor failure handling durable: record `lastAttempts` before checks, persist failure state in a `finally` path, and quarantine/reset malformed state files instead of failing hard.
- Trace and disable the code path that keeps attempting protected heartbeat config patches.
- Treat model-heavy cron staggering/backoff as the main reliability project, with separate isolation for the financial-advisor session reuse that triggered lock timeouts.
